# WorkTrace 详细设计

## 1. 目标与范围

WorkTrace 的目标是每天从自己参与过讨论的飞书会话中提取与工作相关的聊天内容，借助 Codex 做语义分析，生成事项级工作事件，并将结果写入本地 Excel 文件，方便后续回顾、总结和整理日报素材。

首版仅支持手动指定日期执行，重点是把单日处理链路跑通，不实现真正的定时调度。架构设计需要兼顾后续扩展，便于逐步增加自动调度、更多聊天源、更多存储后端以及其他 Agent 封装。

## 2. 设计原则

- 保守高准确，宁可少记也不误记。
- LLM 只负责语义理解，不负责确定性流程控制和数据计算。
- 原始聊天内容不长期落盘，只保留结构化事件结果。
- 仓库根目录同时作为通用脚本项目根目录和 Codex skill 根目录。
- 所有可替换能力通过抽象工厂统一管理，便于后续扩展实现。

## 3. 已确认需求

- 分析范围限定为“当天自己发过消息的会话”。
- 消息读取策略为先抓取整会话当日消息，再围绕本人发言锚点在本地裁剪上下文。
- 事件粒度为事项级，不按单条消息或单次发言分别存储。
- 同一事项允许跨多个会话合并。
- 输出结果只保留日期，不保留时间。
- 与工作无关的聊天内容需要自动忽略。
- `result` 字段允许为空字符串，用于表示当天只有推进过程、没有明确结果。
- Excel 采用单工作簿全年累计，一行对应一条事件。
- 同一天重复执行采用覆盖策略，而不是追加或保留多版本。
- 首版最小输出字段为 `date`、`event_id`、`topic`、`content`、`result`。

## 4. 总体架构

### 4.1 五层结构

WorkTrace 首版采用五层结构：

1. Codex Skill 层  
   负责接收用户在 Codex 对话中的触发请求，解析目标日期，调用底层脚本完成处理，并向用户返回执行摘要。

2. Runner / Orchestrator 层  
   负责组织单日执行流程，串联聊天抓取、上下文裁剪、批量分析、结果校验和存储写入。

3. Source / Resolver / Analyzer / Store 抽象层  
   定义统一接口，通过抽象工厂创建具体实现，降低飞书、Codex、Excel 等组件的耦合。

4. Python 实现层  
   实现首版具体能力，包括飞书消息抓取、消息预处理、附件解析、批量任务组装、Excel 写入和异常处理。

5. 外部依赖层  
   首版依赖 `lark-cli`、Codex 和本地 Excel 文件。

### 4.2 仓库结构建议

建议仓库逐步演进为如下结构：

```text
.
├── SKILL.md
├── README.md
├── docs/
│   └── detailed-design.md
├── src/
└── tests/
```

目录职责约定如下：

- `SKILL.md`：位于仓库根目录，作为 Codex skill 入口说明。
- `src/`：核心通用逻辑，供 Codex skill 和未来其他入口复用。
- `tests/`：单元测试与集成测试。
- `docs/`：设计和架构文档。

### 4.3 Skill 与脚本的关系

仓库根目录就是 skill 根目录，因此安装时应把整个仓库链接到 `~/.codex/skills/<skill-name>`，而不是只链接一个子目录。

这种结构下，根目录 `SKILL.md` 可以直接约束 Codex 在仓库内调用 `src/` 中的脚本入口。推荐做法是通过稳定的 Python 入口命令调用核心逻辑，而不是在 `SKILL.md` 中堆业务实现细节。

## 5. 核心职责分工

### 5.1 脚本职责

Python 脚本负责所有确定性工作，包括：

- 读取目标日期参数。
- 调用 `lark-cli` 抓取当天消息。
- 获取本人身份信息。
- 识别本人发言锚点。
- 裁剪与合并上下文窗口。
- 预处理文本、卡片、链接和附件元数据。
- 在必要时补读附件文本。
- 按批次组织分析输入。
- 调用 Codex 执行批量分析。
- 校验 Codex 返回的 JSON 结构。
- 生成稳定的 `event_id`。
- 覆盖写入 Excel。
- 控制日志、异常和重试。

### 5.2 LLM 职责

Codex 只负责非确定性的语义工作，包括：

- 判断内容是否属于工作事项。
- 将聊天内容归并成事项级事件。
- 提炼事件主题。
- 总结事件内容。
- 提炼事件结果，结果允许为空。
- 识别跨会话是否属于同一事项并合并。
- 忽略与工作无关的内容。

### 5.3 LLM 不负责的内容

Codex 不负责以下工作：

- 数据计算。
- 时间范围筛选。
- 去重逻辑。
- 身份识别。
- Excel 读写。
- 覆盖策略。
- 稳定 ID 生成。

## 6. 抽象工厂与接口

首版设计四类核心接口和一个编排器：

```text
ChatSource
- fetch_day_messages(target_date, self_identity) -> list[NormalizedMessage]

ContentResolver
- to_text(message) -> str
- load_attachment_text_if_needed(message, hint) -> str | None

Analyzer
- analyze_batch(target_date, batch_input) -> list[WorkEvent]
- merge_batch_results(target_date, batch_results) -> list[WorkEvent]

EventStore
- replace_day(target_date, events) -> StoreWriteResult
- read_day(target_date) -> list[WorkEvent]

DailyTraceRunner
- run(target_date) -> DailyRunResult
```

### 6.1 首版实现

首版具体实现规划如下：

- `FeishuCliChatSource`
- `FeishuMessageContentResolver`
- `CodexAnalyzer`
- `ExcelEventStore`

### 6.2 工厂职责

建议使用抽象工厂统一实例创建，至少包括：

- `ChatSourceFactory`
- `ContentResolverFactory`
- `AnalyzerFactory`
- `StorageFactory`

Runner 只依赖接口，不直接依赖实现类，便于未来扩展到其他聊天工具、其他 Agent 和其他存储形式。

## 7. 数据模型

首版至少定义以下数据模型：

### 7.1 `SelfIdentity`

表示当前飞书用户身份，至少包含本人 `open_id`、显示名和身份来源信息，用于识别本人发言锚点。

### 7.2 `NormalizedMessage`

表示统一格式的消息对象，屏蔽 `lark-cli` 原始输出差异。建议包含：

- 会话标识
- 会话名称
- 消息标识
- 发送人标识
- 发送时间
- 消息类型
- 文本内容
- 附件元数据

### 7.3 `ConversationSlice`

表示围绕本人发言锚点裁剪后的上下文切片，用于作为 LLM 的基本分析单元。一个切片可包含多个连续消息，并保留来源会话信息。

### 7.4 `AnalysisBatch`

表示一批待分析的切片集合，用于批量调用 Codex。该对象还应包含批次编号、估算 token 大小和重试上下文。

### 7.5 `WorkEvent`

表示最终结构化工作事件，首版字段固定为：

- `date`
- `event_id`
- `topic`
- `content`
- `result`

补充约束：

- `date` 仅保留 `YYYY-MM-DD`。
- `result` 可为空字符串。
- `event_id` 由脚本稳定生成。
- 不长期保存原始消息正文。

### 7.6 `DailyRunResult`

表示单日执行摘要，建议包含：

- 目标日期
- 会话数
- 原始消息数
- 候选切片数
- 分析批次数
- 生成事件数
- 执行状态
- 错误摘要

## 8. 单日处理流程

单日执行流程按以下顺序进行：

1. 接收目标日期。
2. 获取当前飞书 user 身份。
3. 找出当天自己参与过的会话。
4. 拉取这些会话的当日消息。
5. 根据本人发言定位锚点。
6. 裁剪并合并上下文窗口。
7. 将窗口整理为候选事项切片。
8. 本地粗过滤明显非工作内容。
9. 按 token 预算拆成多个分析批次。
10. 批量调用 Codex。
11. 汇总批次结果并做本地校验。
12. 生成稳定 `event_id`。
13. 删除 Excel 中该日期旧记录。
14. 写入新事件。
15. 输出运行摘要。

## 9. 批量调用策略

首版采用“按天分批、按 token 预算切批”的调用策略。

### 9.1 分批原则

- 不按单条消息调用。
- 不按单条事件调用。
- 一批包含多个 `ConversationSlice`。
- 当某个切片内容过长时，可单独成批。

### 9.2 控制职责

Python 负责：

- 切片分批。
- token 预算估算。
- 批次重试。
- 多批结果汇总。
- 最终本地校验。

Codex 负责：

- 接收一个批次的多个切片。
- 返回结构化 JSON 事件数组。

### 9.3 失败恢复

- 单批失败只重试该批，不重跑整天。
- 整日执行失败时不写入半成品结果。
- 当批次结果格式不合法时，应在 Python 层判定失败并进入重试或终止流程。

## 10. 附件处理策略

- 默认只使用可直接读取的消息文本、卡片文本、链接标题等内容。
- 如果脚本判断 LLM 需要附件内容才能正确总结事件或结果，则补读附件文本。
- 首版不做图片 OCR。
- 附件解析失败不应导致整日失败，应降级为忽略该附件并记录日志。

## 11. 存储设计

首版存储采用单工作簿全年累计模式，并使用一张明细表保存事件。

### 11.1 明细表结构

列顺序固定为：

- `date`
- `event_id`
- `topic`
- `content`
- `result`

### 11.2 覆盖策略

- 同日期重跑时，先删除该日期旧记录，再写入新记录。
- 不保留原始聊天长期归档。
- 不保留版本历史。
- 首版不做汇总表。

## 12. 错误处理与日志

首版至少需要覆盖以下异常场景：

- `lark-cli` 调用失败。
- 身份读取失败。
- 当天无本人发言。
- Codex 单批分析失败。
- JSON 格式不合法。
- Excel 写入失败。
- 附件读取失败。

### 12.1 错误处理要求

- 单批失败可重试。
- 整日失败不应写入半成品。
- 若某个附件读取失败，不影响其他切片继续处理。

### 12.2 日志要求

日志至少应包含：

- 目标日期
- 批次号
- 会话标识
- 当前处理阶段
- 错误摘要

## 13. 扩展方向

以下能力属于后续扩展方向，不在首版范围内：

- 自动定时调度
- Claude Code 封装
- 更多聊天源
- 更多存储后端
- 汇总日报视图
- 候选低置信事件复核机制
