# WorkTrace 部门到中心两级汇总改造说明

> 状态：生产代码和文档已按本方案实现；历史 V1 样本完成确定性审计，真实多人 V2 语义效果仍需后续运行验收。

## 1. 目标流程

```text
员工 V2 个人 MD
  -> 部门负责人运行 merge-collected
  -> YYYY-MM-DD-部门负责人-merged.md
  -> 中心负责人再次运行 merge-collected
  -> YYYY-MM-DD-中心负责人-merged.md
```

两级使用同一个命令：

```bash
python3 -m src.worktrace.cli merge-collected --date YYYY-MM-DD
```

代码不增加部门级或中心级参数。人员收集、文件放置和两次运行的先后顺序由负责人控制；程序只处理当前目录和一级子目录中的各个独立 scope。

预期质量边界：

- 同一真实事项的多个视角合成一个事件
- 不同事项保持分开，不为了减少数量强行合并
- 正文保留各来源不冲突的关键事实，不按人员逐条堆叠
- 一个人组成的部门或当天没有重复事项时，输出数可以等于输入数
- 来源人员、原始事件 ID 和上一级负责人可以逐级追溯

## 2. 人工输入边界

个人 MD、部门 `*-merged.md` 都是合法输入。两者同时存在时：

- 两份文件都会正常进入候选判断
- 程序不比较两者的 `source_event_ids`
- 不拦截，不提示重复来源
- 是否同时放置由负责人根据实际部门结构人工控制

旧 V1 文件缺少同日会话证据，仍会在所有模型调用和文件写入前停止。V1 只能用于解析规模、分批能力和历史输出复盘，不能伪造字段升级为 V2。

## 3. 来源负责人

`WorkEvent` 增加 `source_report_owners`：

- 个人 MD 不从文件名产生来源负责人
- 输入文件是规范 `*-merged.md` 时，从文件名提取上一级负责人
- 事件已有负责人时，与当前文件负责人合并去重
- 分批摘要、正式事件和再次回读持续保留该列表
- 不维护负责人到部门名称的业务映射

中心 Markdown 在列表非空时公开显示：

```text
- 来源负责人: 丁金龙、栗栋
```

旧 V2 Markdown 缺少字段时按空列表读取。个人输入生成的第一级部门结果没有上游负责人时不显示该字段。

## 4. 合并人来源

来源文件姓名与当前飞书登录人完全一致时，标记为合并人来源。

- 没有明确事实冲突时，整合所有来源的互补事实
- 只有版本、结论、状态、结果或待办方向明确冲突时，采用合并人版本并记录说明
- 没有当前登录人的个人 MD 时，直接执行普通汇总，不写 warning
- 输出文件始终以当前飞书登录人命名

## 5. 模型协议与 Python 校验

候选分组返回：

```json
{
  "split_reason": "拆组时说明整体分组差异，未拆组时为空",
  "merged_groups": [
    {
      "group_id": "group-001",
      "draft_ids": ["draft-001", "draft-002"],
      "semantic_reasons": ["same_object"],
      "reason_detail": "说明整个组为何属于同一真实事项",
      "member_connections": [
        {
          "draft_id": "draft-001",
          "connection_detail": "说明该事件与共同对象、动作链或交付批次的直接关系"
        },
        {
          "draft_id": "draft-002",
          "connection_detail": "说明该事件与共同对象、动作链或交付批次的直接关系"
        }
      ],
      "risk_flags": ["broad_object"]
    }
  ],
  "singleton_draft_ids": []
}
```

新模型输出不再返回 `evidence_relation_ids` 或内部 `group_reason`。模型仍能看到 Python 编号的 `MSG-xxx`、`FILE-xxx` 证据目录，但只负责语义判断和逐事件说明；Python 在模型分组后排除端点越界的关系，并按稳定目录顺序计算能够连接全组的最小证据集合。内部 `evidence_relation_ids` 只保存 Python 计算结果并兼容旧 trace。局部证据不能作为全组合并依据。

`member_connections` 必须与每个多事件组的 `draft_ids` 完全一致，不能遗漏、重复、引用组外编号或留空。重复编号和单成员合并组不静默修复，仍作为当前请求的结果质量错误进入局部重试。

正式正文返回：

```json
{
  "covered_draft_ids": ["draft_id"],
  "fact_items": [
    {
      "text": "需要保留的关键事实",
      "source_draft_ids": ["draft_id"]
    }
  ]
}
```

Python 检查每个输入 draft 只出现一次、锁定组未改变、`covered_draft_ids` 完整一致、事实来源只引用当前组且覆盖全部来源。正文覆盖不完整时只重试当前内容组；重试后仍失败则当前 scope 不写文件。

来源负责人相同不是合并依据。单条事件组直接保留原事件，不调用正式正文模型。

## 6. 高风险复核

`config/collected_merge.json` 默认配置：

```json
{
  "high_risk_review_enabled": true,
  "high_risk_source_event_count": 10,
  "high_risk_source_file_count": 4,
  "review_cross_batch_groups": true,
  "review_repaired_groups": true,
  "review_workstream_conflicts": true,
  "review_same_conversation_only_groups": true,
  "review_semantic_only_object_conflicts": true,
  "review_broad_object_groups": true
}
```

共同消息、共同文件、同一对象、连续动作和同批交付材料等规则由同文件中的 `group_reason_definitions` 定义。每项包含 `acceptance_rules` 和 `rejection_rules`：语义理由必须说明具体共同对象、动作承接关系或同一提交任务，并逐条覆盖全部成员；只有地区、部门、系统、业务类别、负责人、标题、时间、文件格式或报表类别相似时不得合并。共同消息和共同文件必须由 Python 计算并连接整个组；同一会话永远不能单独支持合并。完整中文说明只维护在 JSON 配置中，不在 Python 代码中硬编码业务关键词。

单条候选直接保留，不进入高风险复核。多条候选在来源事件达到 10 条、来源文件达到 4 个、跨批再次判断、Python 修复、不同非空工作流、同一会话连接了多个没有共同消息或共同文件的部分、没有完整共同证据且标准化后的非空 `object_hint` 存在多个值，或模型返回 `broad_object` 时，在正式正文前最多三路并行复核。对象冲突只触发针对性复核，不直接自动拆组。跨批仅协调存在共同消息指纹或共同文件的候选。候选分组与高风险复核使用独立 Function 示例，复核示例采用保守拆分，不预填原候选组或 `same_object`。复核可以保留原组或拆成多个子组；同一会话复核保留多事件子组时，必须有 Python 计算的完整共同消息/文件证据，或使用配置允许的语义依据，不能只用同一会话。拆开时只需一条顶层 `split_reason` 说明整体分组差异，旧格式中任一子组已有理由也可接受；完全没有理由时 Python 才保留原组并记录告警。结果必须完整覆盖原组，不得遗漏、重复或增加 draft ID。

分组协议错误会把错误组、重复编号、证据端点和缺失成员反馈给模型，并明确要求补入缺失成员或删除该合并。结果质量错误固定走 Online 首次请求、Online 局部重试 1 次、Codex 当前请求备用 1 次；Codex 仍失败或结果仍不合法时终止本次合并。调试模式只增加 trace，不改变线路和次数。

候选、复核和正式内容请求使用任务专用 Function，并以 `model_input_batch_target_tokens=5200` 为模型输入估算目标。统一估算取“最终提示词、合法参数示例、证据编号、重试反馈和 `/no_think` 加完整 Function 与 `tool_choice`”同“相同提示词加 Codex output-schema”两者的较大值。复核超过目标时先按消息、文件和会话关系分批；单条正文仍过长时复用正文切片和分层摘要，无法继续拆分的最小输入允许发送。

## 7. 正文处理

`_fill_collected_merge_group_metadata(...)` 不再把全部来源正文追加到模型正文。

- 正文完整性由覆盖 ID、事实来源和有限重试保证
- 标题、具体对象、保留理由和保留依据仍可从来源补齐
- 明确冲突仍按合并人来源处理
- 分层正文摘要会把事实来源映射回原始 draft ID
- 失败时不产生不完整输出

## 8. 质量报告

每个 scope 和整次运行都生成 `CollectedMergeQualitySummary`。内容包括：

- 输入、过滤后和输出事件数
- 多来源组、单条组和单组最大来源数
- 输入/输出正文字符数及两类输出输入比例
- 来源事件覆盖率和来源负责人数量
- 高风险、已复核、被复核拆开的组数
- 正文重试数、提示缩短数和是否需要复核

全部数量和比例由 Python 计算。质量报告进入 CLI stdout JSON、trace `summary.json` 和 `summary.md`。不设置强制减少比例，只供人工检查。

## 9. 历史样本审计

本节数字来自实现阶段的一次临时确定性审计。审计脚本和输出位于被 `.gitignore` 排除的 `tmp/`，未随当前仓库提供，不能作为仓库内可复现入口；当前可复现的验证范围以第 10 节列出的自动测试为准。

脚本检查了 10 个日期、54 个文件、1399 个事件。全部事件都缺少 V2 会话证据，因此没有把历史文件送入当前模型流程。加入新协议字段后，当前关系优先分批对这些正文需要 1 至 6 个候选批次，最大估算 6198 token，且没有分批错误；这说明机械分批可控制在 6200 以内，但不代表语义分组已经通过真实 V2 验收。

历史中心结果还显示过两类问题：部分日期输出正文接近输入原量，说明旧逻辑重复追加来源；2026-07-10 曾有单组包含 25 个来源事件，说明大组存在混入不同事项的风险。本次正文覆盖协议和高风险复核分别处理这两个问题。

## 10. 验证范围

自动测试覆盖：

- 旧 V2 缺少来源负责人字段时兼容读取
- 两次 Markdown 读写不丢来源负责人
- 部门到中心继续保留原始人员、事件 ID 和负责人
- 个人与部门文件并存时不拦截、不提示
- 一个人部门输入输出数量相同
- 9 条不触发数量复核，10 条触发；4 个文件触发
- 单条候选跳过复核；多条跨批、修复和工作流冲突触发复核
- 没有完整证据且对象不一致、`broad_object` 分别触发复核；对象一致或已有完整证据时不新增触发
- Python 排除组外证据端点，并稳定选择消息和文件共同组成的最小完整连接集合
- 逐事件说明遗漏、重复、未知编号和空说明进入结果质量重试
- 高风险复核示例不预填原候选组或 `same_object`
- 有拆分理由的复核拆组、无拆分理由时保留原组，以及无效覆盖失败
- 正文覆盖局部重试及失败不落盘
- 质量统计与 trace JSON、Markdown 摘要一致
- 138 和 195 事件规模的候选分批不超过 6200 token
- 离线回放脚本同时覆盖候选分组和高风险复核，并确认 `model_call_count=0`

后续真实 V2 验收应选取多人描述同一事项、同一会话不同事项、相似标题不同对象和一个人部门四类日期。验收重点是误合并、漏合并、关键事实遗漏和高风险复核拆组结果，不以压缩比例作为唯一结论。
